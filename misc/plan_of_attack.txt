GOAL:
Improve performance of a Postgres DB management app by:
1. Avoiding loading BYTEA (binary) data in normal queries
2. Adding connection pooling / caching
3. Reducing redundant metadata queries
4. Supporting lazy-loading of BYTEA via download links

---

CONTEXT:
The app currently:
- Uses SELECT * on tables (including BYTEA columns)
- Recreates the DB engine on every request
- Calls metadata functions repeatedly
- Loads binary data unnecessarily over the network

This works locally but is very slow with a remote database.

---

REQUIRED CHANGES:

========================
1. EXCLUDE BYTEA FROM MAIN QUERIES
========================

Problem:
- SELECT * loads large binary columns (BYTEA), causing major slowdown.

Solution:
- Dynamically fetch column types
- Exclude BYTEA columns from SELECT
- Replace BYTEA column values with a download link placeholder

Implementation:

In get_rows():
1. Call get_column_types()
2. Build a list of columns excluding BYTEA
3. Use SELECT col1, col2, col3 instead of SELECT *
4. Keep track of which columns are BYTEA

Then, after fetching rows:
- For each BYTEA column in each row:
    Replace value with a string like:
    "Download"

    or ideally:
    a link to:
    /api/blob/<table>/<id>/<column>

NOTE:
- Assume tables have a primary key "id"
- If not, infer or adapt accordingly (is this best to ask for the id column for the user to select?)

---

========================
2. ADD BYTEA DOWNLOAD ENDPOINT
========================

Goal:
- Load binary data ONLY when explicitly requested

Add new route:

/api/blob/<table>/<id>/<column>

Behavior:
- Query:
    SELECT column FROM table WHERE id = :id
- Return raw binary data
- Set headers so browser downloads file:
    Content-Disposition: attachment

Do NOT:
- Cache this
- Store in memory longer than needed

---

========================
3. CACHE DATABASE ENGINE (CONNECTION POOLING)
========================

Problem:
- Engine is recreated on every query
- No connection reuse

Solution:
- Cache engine using LRU cache

Implementation:

Wrap get_engine() with:

@lru_cache(maxsize=10)

This ensures:
- Connection pooling is actually used
- Remote DB latency is reduced

---

========================
4. CACHE COLUMN METADATA
========================

Problem:
- get_column_types() is called repeatedly

Solution:
- Cache results

Implementation:

@lru_cache(maxsize=100)
def get_column_types(db_conn_string, table):

---

========================
5. AVOID DUPLICATE METADATA CALLS
========================

Within get_rows():
- Call get_column_types() ONCE
- Reuse result everywhere (filtering, sorting, etc.)

---

========================
6. SKIP BYTEA IN FILTERING
========================

Problem:
- Filtering casts all columns to TEXT
- BYTEA should not be included

Solution:
- When building filters:
    skip columns where type == "bytea"

---

========================
7. (OPTIONAL) UI CHANGE
========================

For BYTEA cells:
- Show:
    "Download"
- Make it a clickable link to:
/api/blob/<table>/<id>/<column>

---

EXPECTED RESULT:
- Dramatically faster queries on remote DB
- No large binary transfers unless explicitly requested
- Reduced latency due to connection reuse
- Cleaner and more scalable architecture

---

IMPORTANT CONSTRAINTS:
- Do NOT remove BYTEA support entirely
- Do NOT preload or cache binary data
- Only fetch BYTEA on-demand via download endpoint
- Keep changes minimal and consistent with existing structure

---

DELIVERABLE:
- Updated backend code implementing all above
- Minimal changes to frontend to support download links
